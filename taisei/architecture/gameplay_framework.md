# 遊戲邏輯框架架構

Taisei Project 的遊戲內容邏輯圍繞實體、協程和自定義 DSL 構建，這允許開發者編寫清晰的、命令式的複雜彈幕腳本。

## 實體組件系統 (`src/entity.h`)

雖然不是現代意義上的完整 ECS (Entity-Component-System)，但 Taisei 對遊戲對象使用了結構化的方法：
-   **通用實體接口**：所有遊戲對象（玩家、敵人、彈幕、物品、激光）共享一組基本屬性和生命週期回調。
-   **內存高效池**：許多實體都在專門的池中管理 (`src/stageobjects.c`)，以避免遊戲期間頻繁的分配和釋放。
-   **實體列表**：`Global` 結構體包含每種類型的實體列表，允許高效的更新和碰撞檢測。

## 基於協程的邏輯 (`src/coroutine/`)

遊戲邏輯的核心由協程處理，由 `koishi` 庫提供支持。這允許開發者編寫可以跨幀暫停和恢復的代碼，使複雜的定時邏輯易於實現。
-   **任務 (`CoTask`)**：協程組織為任務。每個任務都有自己的堆棧，並可以將控制權交還給引擎。
-   **暫停和等待**：`WAIT(n)` 等宏允許任務暫停 `n` 幀後繼續。
-   **綁定到實體**：任務可以「綁定」到特定的遊戲實體。如果實體被銷毀，任務也會自動取消。

## 任務 DSL (`src/coroutine/taskdsl.h`)

通過 C 宏實現的強大領域特定語言 (DSL) 允許進行高級腳本編寫：
-   **`TASK(name, args)`**：定義一個帶有參數的新協程任務。
-   **`INVOKE_TASK(name, ...)`**：啟動一個新任務。
-   **`PROJECTILE(...)`**：使用簡潔的、類似構建器的語法生成彈幕。
-   **`MOVE_...`**：可以組合的高級移動函數（線性、漸近、正弦等）。

## 關卡和時間軸 (`src/stages/`)

每個遊戲關卡都組織在自己的目錄中：
-   **`timeline.c`**：關卡的「主」腳本，使用任務 DSL 生成敵人和觸發事件。
-   **`spells/` 和 `nonspells/`**：BOSS 的特定攻擊模式，通常實現為一個或多個協程任務。
-   **背景**：專門的代碼和著色器，用於渲染每個關卡的 3D 背景和視覺效果。

## 碰撞和物理

-   **基於圓形的碰撞**：大多數遊戲對象使用圓對圓或圓對點的碰撞檢測，這對於處理大量對象非常高效。
-   **移動邏輯 (`src/move.c`)**：靈活的移動系統允許實體根據速度、加速度、吸引點和自定義函數遵循複雜的路徑。
-   **擦彈 (Graze) 系統**：引擎包含「擦彈」機制（東方風格遊戲的標誌），用於檢測子彈何時靠近玩家但未實際擊中。
