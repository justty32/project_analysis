# 實體子類分析：13_entity_subclass_player (玩家實體)

`Player` 是引擎中最複雜的實體，它不僅是物理對象，還承載了大量的持久化遊戲狀態。

## 1. 核心組件與子系統
玩家實體通過多個專用組件來解耦邏輯：
- **PlayerInventory (背包系統)：** 
    - 管理數百個道具槽位，處理道具的交換、堆疊邏輯與過濾查詢。
    - 持有道具的 `ItemDescriptor`，確保在網絡與磁碟上的輕量級序列化。
- **StatusController (屬性控制器)：** 
    - 管理生命值、能量、飢餓度等動態數值。
    - 處理來自 `Item` 或環境的屬性修改器（Stat Modifiers）。
- **TechController (科技控制器)：** 
    - 處理玩家的特殊能力（如雙段跳、衝刺）。
    - 提供 Lua 腳本回調接口，允許 Mod 動態定義能力邏輯。

## 2. Humanoid 動畫系統
玩家使用 `Humanoid` 動畫模型，這是一種基於身體部位（Head, Chest, Arms, Legs）的組合動畫。
- **組件式渲染：** 玩家的視覺外觀是由盔甲、膚色與髮型實時組合而成的，利用 `NetworkedAnimator` 同步各部位的狀態。
- **狀態機切換：** 根據物理狀態（如 `Idle`, `Walk`, `Jump`, `Swim`）自動切換動畫幀集。

## 3. 網絡同步模型 (Player Sync)
- **客戶端權威 (Client Authority)：** 為了消除移動延遲，玩家的物理移動與動作在本地是權威的。服務器僅進行粗略的合法性檢查。
- **Delta 更新：** 玩家背包的大部分變更（如移動道具）僅在需要時（如打開容器或與他人交易）才進行全量同步，其餘時間採用增量更新。

## 4. 身份與 UUID
每個玩家由一個唯一的 `Uuid` 標識。這個 UUID 是其在全宇宙中的唯一 Key，關聯其 `.player` 存檔與對應的飛船 `ShipWorld`。
