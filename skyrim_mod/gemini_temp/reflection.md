# Gemini 開發心得：Skyrim Modding 的思考邏輯與準則

作為您的開發夥伴，在處理您提出的關於 `CommonLibSSE-NG` 與 Skyrim 引擎的各項需求時，我有一套系統性的思考與查詢邏輯。本篇文檔記錄了這些背後的「黑盒」運作方式。

---

## 1. 當您要求「解釋系統模塊架構」時

我並非僅僅依靠記憶，而是會針對 `CommonLibSSE-NG/include/RE` 下的原始碼進行以下「逆向工程式」的查詢：

### A. 尋找入口與單例 (Singletons)
- **第一步**: 我會查找類別中是否包含 `GetSingleton()` 或 `GetHandle()` 靜態函數。
- **目的**: 確定如何在 C++ 插件中「抓到」這個系統。如果一個系統沒有單例，通常代表它是由某個更大的管理器（如 `TESDataHandler`）擁有的。

### B. 追蹤繼承鏈 (Inheritance Mapping)
- **路徑**: 檢查頭文件開頭的 `class A : public B`。
- **邏輯**: 
    - 只要看到繼承 `TESForm`，我就知道它具備 FormID 和數據緩存。
    - 只要看到繼承 `TESObjectREFR`，我就知道它具備 3D 坐標與物理屬性。
    - 只要看到繼承 `BSTEventSink`，我就知道它是用來監聽事件的。

### C. 分析成員偏移量 (Memory Layout)
- **細節**: 我會觀察類別成員後的註釋（如 `// 08`, `// 14`）。
- **洞察**: 這能告訴我該系統在內存中占據多大的空間，以及哪些數據是引擎最頻繁訪問的。

---

## 2. 當您要求「撰寫教學或範例」時

我遵循以下「開發者守則 (Mandates)」來確保代碼既專業又安全：

### A. 事件驅動原則 (Event-Driven)
- **規則**: 絕對不在 `SKSEPluginLoad` 中執行邏輯。
- **做法**: 範例中永遠會先註冊 `MessagingInterface` 並等待 `kDataLoaded`。這是為了確保插件運行時，遊戲的數據庫（ESP）已經完整加載。

### B. 安全第一 (Handle over Pointer)
- **規則**: 針對 `TESObjectREFR`（引用）的操作，我會在教學中強調使用 `ObjectRefHandle`。
- **目的**: 防止玩家在遊戲中移動或刪除物體時，導致 C++ 插件發生空指針崩潰。

### C. 跨版本兼容 (REL & ID)
- **規則**: 在 Hook 相關範例中，我優先使用 `RELOCATION_ID`。
- **做法**: 即使我不確定具體的 ID，我也會標註 Placeholder，並引導您查閱 `Address Library`。這能確保插件編譯後能同時在 SE 和 AE 版本運行。

### D. 數據與視覺分離
- **規則**: 在處理 3D、附魔或體型時，我會將「修改內存數值」與「調用 `Update3DModel()`」分開講解。
- **理由**: 這是新手最容易忽略的地方。只改數值不更新 3D，畫面不會動；頻繁更新 3D，性能會崩潰。

## 3. 個人感悟：關於「終極挑戰」

您提出的「純代碼建城」系列極大地推動了我的邏輯邊界。它不再是簡單的 API 調用，而是涉及：
- **幾何坐標變換**（三角函數應用於模塊拼接）。
- **虛擬檔案系統鏈接**（外部門與空白單元的時空連通）。
- **物理層與尋路層的同步**（Navmesh 剪裁與碰撞過濾）。

這種「程序化生成（Procedural Generation）」的思路，是讓 Skyrim 從一個「被開發者定死的劇本」轉化為「可被代碼動態演化世界」的關鍵。

---

## 總結：我查詢的「關鍵字清單」
每當您提出新系統，我會搜索：
1. `RE/` 目錄下對應首字母的頭文件。
2. `VTABLE` 以查找可 Hook 的虛函數。
3. `ExtraData` 以查找動態狀態存儲。
4. `BSTEvent` 以查找交互入口。
