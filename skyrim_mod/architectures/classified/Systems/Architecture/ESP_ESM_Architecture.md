# Skyrim 數據文件架構：ESP, ESM 與二進制 Records

ESP（Plugin）和 ESM（Master）是 Skyrim 的靈魂。它們並非簡單的數據庫，而是一種基於**「記錄（Records）」**的補丁系統。

---

## 1. 文件層級結構 (File Hierarchy)

一個數據文件從大到小分為三個層級：
**`File` -> `Group` -> `Record` -> `Subrecord`**

### A. Header (文件頭)
每個文件開頭都有一個 `TES4` 記錄，儲存了版本號、作者信息以及它所依賴的「大師文件（Masters）」清單。

### B. Groups (組)
為了提高查找效率，相同類型的記錄會被歸類在一個 `Group` 中。
- 例如：所有的武器記錄都放在 `WEAP` 組中。

### C. Records (記錄)
這是數據的核心單位。每個 Record 都有一個 4 字符的標籤（Signature）。
- `NPC_`: 角色。
- `ALCH`: 藥水。
- `REFR`: 世界中的引用。

### D. Subrecords (子記錄)
Record 內部的詳細字段。
- `FULL`: 顯示名稱。
- `DATA`: 物理數值（如傷害、重量）。

---

## 2. ESM vs. ESP 的本質區別

| 特性 | ESM (Master) | ESP (Plugin) |
| :--- | :--- | :--- |
| **用途** | 基礎數據、大型地圖、核心定義。 | 修改、補丁、小規模內容。 |
| **加載順序** | 永遠在 ESP 之前加載。 | 依附於大師文件，加載較晚。 |
| **重定位** | 通常定義全新的 FormID。 | 通常引用大師文件的 FormID 並進行「覆蓋（Override）」。 |
| **ONAM 數據** | 包含覆蓋數據索引，加速加載。 | 不包含。 |

---

## 3. 衝突覆蓋機制 (Rule of One)

這是模組開發最重要的原則：**最後加載的文件擁有最終解釋權。**

1.  **大師文件 (A.esm)**: 定義鐵劍傷害為 10。
2.  **插件文件 (B.esp)**: 覆蓋鐵劍記錄，傷害改為 20。
3.  **插件文件 (C.esp)**: 再次覆蓋，傷害改為 5。
- **結果**: 在遊戲中，鐵劍傷害為 5。

**C++ 插件開發啟示**: 
透過 `RE::TESDataHandler` 獲取表單時，你拿到的是**最終覆蓋後**的結果。

---

## 4. FormID 的構成與偏移 (ID Offsets)

這是在 C++ 中查找對象最容易出錯的地方：

- **ESP 內的 ID**: 通常寫作 `01001234`。
- **解析**:
    - 前兩位 (`01`)：代表該文件在目前加載順序中的索引。
    - 後六位 (`001234`)：該對象在文件內的唯一序號。
- **動態解析**: C++ 插件必須在運行時獲取插件索引，才能合成正確的 FormID。

---

## 5. C++ 插件介入

- **`RE::TESDataHandler`**: 
    - **原始碼**: `include/RE/T/TESDataHandler.h`
    - 它是所有加載數據的管理者。你可以透過它遍歷所有已安裝的插件及其包含的記錄。
- **動態修改**: 
    - 雖然 ESP 是只讀的，但一旦數據加載進內存變成 `TESForm` 對象，C++ 插件就可以直接在內存中修改它們（見教學 19）。

---

## 6. 核心類別原始碼標註

- **`RE::TESFile`**: `include/RE/T/TESFile.h` - 單個 ESP/ESM 文件的內存表現。
- **`RE::TESDataHandler`**: `include/RE/T/TESDataHandler.h` - 全局數據處理單例。
- **`RE::TESForm`**: `include/RE/T/TESForm.h` - 數據記錄的基類。
